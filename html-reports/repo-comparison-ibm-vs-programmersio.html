1TL;DR

Programmersio-IBMi/ia-mcp-ibmi is a hard fork of IBM/ibmi-mcp-server v0.5.1. It is not a GitHub fork — the parent field is null and the initial commit is a clean import. Twelve commits over ten days (May 15–25, 2026) transform it into a Programmers.io-branded commercial product.

Three substantive changes ride on top of the upstream code:

  1. Rebranding from @ibm/ibmi-mcp-server → @programmersio-ibmi/ia-mcp-server (mechanical, ~35 files).
  2. License enforcement — a runtime gate that polls a customer-hosted validator, blocks MCP requests with a branded JSON-RPC -32001 error when the license is invalid, and never exits the process (~1,500 LOC added across two source files, two test files, mock validator, and wiring).
  3. Bundled iA tooling — IBM's example YAML toolsets (developer / performance / security / sys-admin / sample) are removed, replaced with a single tools/impact-analysis.yaml (104 KB, 49 read-only SQL tools for Impact Analysis on IBM i).

Plus: distribution moves from public npm to private GitHub Packages, an Apache-2.0 NOTICE is added for attribution, and the upstream-merge workflow is retargeted to IBM/ibmi-mcp-server with a merge=ours driver to protect branded paths.

3Relationship: fork or hard copy?

The GitHub API reports parent: null and source: null for the iA repo — it is not a GitHub-managed fork. Instead, the contents of IBM/ibmi-mcp-server at v0.5.1 were committed as a single initial commit by user khushi-pio on 2026-05-15. This buys editorial freedom (rename freely, rewrite history, never expose the fork relationship in GitHub's UI) at the cost of losing GitHub's automatic "X commits behind upstream" tracking.

Upstream sync is handled instead by a custom .github/workflows/upstream-merge.yml retargeted at IBM/ibmi-mcp-server, with a merge=ours git attribute on branded paths (package.json files, NOTICE, README intro) so future upstream merges don't clobber the rebrand.

Why this matters: Both repos are Apache-2.0. The fork is a legitimate downstream — it complies with §4 of the license (attribution preserved in NOTICE, modifications listed in CHANGELOG, source available to anyone with a PAT). The hard-fork model is a packaging/branding decision, not a license maneuver.

6License enforcement — the headline feature

This is the only behavioral difference between the two servers. Everything else is packaging. The implementation is intentionally small and trust-based:

Architecture

  • Validator is customer-hosted. IA_LICENSE_VALIDATOR_URL points at an endpoint on the customer's iA installation (e.g. http://<ia-host>:8080/iaTool/api/aiLicenseValidator). No JWT, no signing — the validator returns the plain string "true" or "false".
  • Server boots regardless. initializeLicenseGate() runs once at startup; the result sets a module-level flag _currentlyValid. Even on false, the process stays alive and listens — only MCP requests are gated.
  • Polling is adaptive. setTimeout(tick, 1h) when valid, setTimeout(tick, 5min) when blocked. State transitions are logged.
  • Grace cache. If the validator is unreachable, the module falls back to a last-good result written to ~/.cache/ia-license.json, but only if it's less than 24 h old.
  • Hono middleware short-circuits with HTTP 403 and a JSON-RPC -32001 error envelope when blocked. Mounted as step 4 in httpTransport.ts (before content-type / auth / rate-limit). /healthz is excluded so liveness probes still pass.
  • Testable in isolation. The module uses dependency-injection seams (setLicenseLogger, LicenseGateState) so the 25 unit tests don't need to boot the full server.

Key code paths

// packages/server/src/license/check.ts (excerpt)
const VALIDATOR_URL = process.env.IA_LICENSE_VALIDATOR_URL;
if (!VALIDATOR_URL) {
  throw new Error(
    "IA_LICENSE_VALIDATOR_URL is required. Set it in .env, e.g., " +
      "http://<ia-host>:8080/iaTool/api/aiLicenseValidator",
  );
}

async function checkValidator(): Promise<boolean | null> {
  try {
    const resp = await fetch(VALIDATOR_URL!, {
      signal: AbortSignal.timeout(FETCH_TIMEOUT_MS),
    });
    if (!resp.ok) return false;
    return (await resp.text()).trim().toLowerCase() === "true";
  } catch (err) {
    log.error({ err }, "License validator unreachable");
    return null;        // unreachable → fall back to cache
  }
}
// packages/server/src/license/middleware.ts (full)
export function createLicenseGate(state: LicenseGateState): MiddlewareHandler {
  return async function licenseGate(c: Context, next: Next) {
    if (!state.isValid()) {
      return c.json(
        {
          jsonrpc: "2.0",
          id: null,
          error: { code: -32001, message: state.message() },
        },
        403,
      );
    }
    await next();
  };
}
Design choice: Trust-based, run-but-block. The validator returns a boolean — there's no cryptographic proof. The author flags this explicitly in the file header. The intent is operational gating (a customer who stops paying loses access on the next 5-min poll), not theft prevention against a determined adversary who controls the network or the process.

    • where-used
    • call hierarchy

The 0.6.1 patch (d0bfd03) discovered that the published 0.6.0 tarball didn't actually contain this YAML — "files": ["dist"] excluded it. The fix adds a prepack script (scripts/bundle-files.ts) that copies tools/impact-analysis.yaml from the monorepo root into packages/server/tools/ before npm publish, then exposes it via package.json exports so customers can resolve it with require.resolve('@programmersio-ibmi/ia-mcp-server/tools/impact-analysis.yaml').

10Bottom line

IBM/ibmi-mcp-server is the platform: an open-source, generic MCP server with example toolsets and a tutorial, intended to be configurable by anyone with an IBM i and a YAML editor.

Programmersio-IBMi/ia-mcp-ibmi is the product: the same platform, narrowed to one tool catalog (the iA SQL tools), gated by a license check the upstream doesn't have, and shipped through a private channel keyed to per-customer PATs. Future upstream improvements flow in through a merge=ours-protected sync workflow; future product work happens on top of the license module and the bundled YAML.

The fork's twelve commits map cleanly to that intent: one to copy, one to rename, one to gate, one to bundle, and eight to keep the release pipeline honest.
